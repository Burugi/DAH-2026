# 코드팀 전달 — 버그 리포트 (fleet 크기 축소 시 IndexError)

## 요약
`src/sim/fleet.py`의 `generate_fleet()`가 **합성 공격(`attacks`)의 target 드론 인덱스를 실제 fleet 크기 `n`으로
검증/클램프하지 않아**, fleet을 설정의 최대 target 인덱스보다 작게 만들면 `IndexError`로 크래시한다.

## 재현
- 설정 `src/configs/sweep.yaml`:
  - `fleet: {n_uav: 12, n_ugv: 6}` → n=18 (정상)
  - `attacks: - {type: gps_spoof, targets: [12, 13], ...}` ← 드론 #12·#13 하드코딩
- `n_uav: 8, n_ugv: 4` (n=12)로 줄이면 유효 인덱스 0~11 → target 12·13이 범위 초과.

```
File "src/sim/fleet.py", line 52, in generate_fleet
    pos_rep[t][tgt] = pos[t][tgt] + drift * (k + 1)
IndexError: index 12 is out of bounds for axis 0 with size 12
```
- 동일 위험: 같은 함수 47~48줄 `snr[...][:, tgt]`, `label_jam[...][:, tgt]`도 jam target이 n 초과 시 동일.

## 원인
```python
for atk in attacks or []:
    tgt = list(atk["targets"])          # <- 설정값 그대로, n 검증 없음
    ...
    pos_rep[t][tgt] = pos[t][tgt] + ...
```
target 인덱스가 fleet 크기와 무관하게 하드코딩되어, fleet 축소 시 배열 범위를 벗어난다.

## 수정 (한 줄)
```python
for atk in attacks or []:
    tgt = [i for i in atk["targets"] if i < n]   # 실제 fleet 크기로 필터
    if not tgt:
        continue
    ...
```
(또는 설정 로드시 target을 `min(i, n-1)`로 클램프.)

## 영향
- 기본 설정(n=18)에선 발생 안 함 → 평소 운영엔 무해.
- 단, **fleet 크기 스케일 실험·소형 군집·동적 군집**에서 크래시 → 확장성 테스트를 막음.
- 우리 확장성 실험은 n≥15로 우회해 진행함(레포 미수정). 본선에서 군집 크기를 바꾼다면 선제 수정 권장.

## 참고
- 발견 경위: 코디네이터 방어의 *확장성*(n=15·24·36) 측정 중. 수정 후 n=12 이하도 테스트 가능.
